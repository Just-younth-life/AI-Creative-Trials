# 产品需求文档（PRD）：AI 社交日程管理产品

> 版本：v1.0 | 日期：2026-07-23 | 作者：Queen

---

## 一、执行摘要

本产品线采用 **双版本策略**：
- **公开版**：面向学生自律群体的「社交+AI」日程管理工具，核心差异是"公开承诺驱动的完成率提升"
- **情侣版**：面向情侣的私密协同工具，核心差异是"任务协同+情感陪伴"

两个版本共享底层 AI 能力，但用户池、社交关系、数据完全隔离。优先验证公开版，成功后启动情侣版。

**核心结论**：产品可行，但需要极度聚焦 MVP，逐步验证。

**Strava的关键启示**：
1. **"社交货币"是核心壁垒**：kudos（点赞）、排行榜、segment竞争，把运动数据变成社交互动——用户不只是记录，更是"被看见"

4. **Gen Z驱动增长**：年轻用户把跑步当社交方式，Z世代"用跑步替代约会App"

**对我们的启示**：Strava证明了"社交+效率工具"的商业模式完全成立，且估值极高。但Strava做的是运动场景，**任务管理/日程场景还没有出现"Strava级别"的社交产品**——这正是机会。

| **Flutter双端同时** | 一套代码覆盖两端、不丢失任何用户 | 开发成本略高、部分原生功能需分别适配 | **推荐方案** |

### 2.6 平台选择最终建议：Flutter双端同步，但分阶段侧重

**MVP阶段（0-3个月）：Flutter双端同步上线**

理由：
1. **Flutter一套代码覆盖iOS+Android+鸿蒙（通过适配）**，双端开发成本只比单端高约30%
2. 你的核心差异是"社交问责"——**社交需要网络效应**，如果只有iOS版，70%的目标用户看不到你，冷启动更难
3. 学生群体的社交传播是"宿舍/班级"级别的，一个宿舍里有人用Android有人用iPhone，**必须双端都有**
4. 推送可靠性问题可以通过极光推送/JPush解决

**但资源分配上**：
- iOS端：重点优化**付费流程、订阅体验、小组件/锁屏**（因为iOS用户更可能付费）
- Android端：重点优化**推送到达率、分享链路、小组件兼容性**（因为Android用户是传播主力）

**Phase 2（3-6个月）**：根据数据判断哪端付费率更高，投入更多资源优化。

**特别注意鸿蒙**：
- 鸿蒙已占19%且还在增长，但鸿蒙Next已不再兼容安卓APK
- 如果目标用户中华为手机占比超过20%，需要单独适配鸿蒙
- 建议MVP阶段先用Flutter的Android版本覆盖非华为用户，Phase 2再评估是否独立开发鸿蒙版

## 三、产品一：公开网络社交版

### 3.1 思考过程：为什么这样设计功能

**核心问题**：在滴答清单（全能GTD）、番茄Todo（专注力）、Notion（知识库）已经很强的情况下，一个新产品的差异化在哪？

**推理链条**：

1. **现有产品解决"记录"但不解决"执行"**
   - 用户痛点不是"不知道该做什么"，而是"知道但就是不做"
   - 行为经济学证据：公开承诺将完成率从10-25%提升到65%（StickK数据）
   - **机会**：做"从记录到完成的最后一公里"

2. **"公开"是差异化核心，但需要本土化适配**
   - 中国文化中"面子"是双刃剑：公开承诺既有激励效果，也有"丢脸"恐惧
   - 解法：**选择性公开 + 正向激励为主**（不是惩罚，而是被看见、被鼓励）
   - 参考：小红书的"打卡文化"、朋友圈的"自律人设"——中国用户已经接受了"展示努力"的社交行为

3. **学生是最佳切入点**
   - 对"公开承诺"接受度最高（朋友圈打卡文化已验证）
   - 传播性强（宿舍/班级/社团的口碑效应）
   - 场景明确（考研/考公/考证/期末）
   - 竞品空白：番茄Todo有自习室但没有"目标公开+问责"，滴答清单完全没有社交

4. **AI不是噱头，是"效率倍增器"**
   - 竞品已有AI功能（Todoist Ramble、滴答清单AI），但这些只解决"录入效率"
   - 我们的AI要解决"完成动机"：基于社交信号的个性化nudge
   - 例如："你的3个朋友今天都完成了考研数学，你还差2章"

### 3.2 产品定位

> **一句话定位**：让你的目标"被看见"的AI日程管理工具——通过公开承诺和轻社交问责，帮你真正完成任务。

**核心差异化公式**：
```
公开承诺（动力引擎）+ AI辅助（效率引擎）+ 选择性透明（隐私保障）
= 传统todolist无法复制的体验
```

### 3.3 功能设计与优先级

#### P0 | MVP核心（4-6周开发，验证核心假设）

**核心假设**：公开承诺+轻社交问责能显著提升任务完成率

| 功能模块 | 具体功能 | 设计理由 |
|---------|---------|---------|
| **基础任务管理** | 创建/完成/子任务/日期/优先级（P0-P3）/标签/提醒 | 工具底座，不好用就没有后续 |
| **公开承诺页面** | 用户选择将特定目标/进度公开，生成可分享的承诺页面 | **核心差异化功能**，验证"被看见"是否真的有效 |
| **轻社交互动** | 关注/鼓励（cheer）/简单排行榜（按连续打卡天数） | 不做即时通讯，不做私信，只做"看见+鼓励" |
| **基础AI** | 语音转任务（调系统原生语音输入）+ 基础任务拆解 | 降低录入门槛，不做复杂AI |
| **日历联动** | 待办与日历双向显示，"我的一天"视图 | 用户基础需求，竞品标配 |

**MVP不做**：
- ❌ 番茄钟/白噪音（番茄Todo已做好，不是我们的差异点）
- ❌ 知识库/文档管理（Notion已做好，太重）
- ❌ 邮箱集成（技术复杂度高，用户频次低）
- ❌ 广告系统（先跑通产品，再考虑变现）
- ❌ 情侣版（独立验证主线后再做）

**MVP成功标准**：
- 公开承诺用户的任务完成率 > 非公开用户 50%以上
- D30留存率 > 35%
- 自然传播系数（每个用户带来的新用户数）> 0.3

#### P1 | 增强版（MVP验证后3个月内）

| 功能模块 | 具体功能 | 设计理由 |
|---------|---------|---------|
| **赛道排行榜** | 考研/公考/考证/健身等分赛道，按"连续打卡天数"排名 | 给公开承诺加上竞争激励 |
| **AI进阶** | 日/周复盘生成 + 拖延行为分析 + 智能排程建议 | 提升AI价值感，为付费做铺垫 |
| **专注模式** | 番茄钟 + 专注时长统计 + 锁屏小组件 | 补齐效率工具标配功能 |
| **进度可视化** | 每周/月生成精美的进度报告卡片，可分享到社交平台 | 社交传播引擎 |
| **习惯追踪** | 每日打卡 + 连续天数 + 对关注者可见 | 增加日活和粘性 |

#### P2 | 终局版（6-12个月后）

| 功能模块 | 具体功能 |
|---------|---------|
| **自律社区** | 优质复盘精选、方法论分享、无杂乱信息流 |
| **高阶AI教练** | 个性化学习规划、长期成长路径建议 |
| **精美年度报告** | 网易云音乐风格的年度自律报告，社交传播 |
| **团队版** | 小团队（2-5人）公开看板，面向创业团队 |

### 3.4 核心交互流程

```
用户注册 → 创建第一个目标（如"30天考研数学第一章"）
  → 选择是否公开（默认私密，用户主动选择公开）
  → 公开后进入赛道排行榜
  → 每日完成任务后打卡
  → 关注者收到通知 → 鼓励/cheer
  → AI每日/周生成复盘
  → 用户分享进度卡片到社交平台
  → 新用户被吸引注册 → 循环
```

**关键交互原则**：
1. **创建任务 ≤ 3步**：语音/文字输入 → AI自动拆解 → 确认
2. **公开是主动选择**：默认私密，用户可随时切换公开/私密
3. **社交是轻量正向的**：只有鼓励（cheer），没有批评；只有打卡天数排名，没有"谁最差"
4. **AI是辅助不是主导**：AI建议可以一键接受或忽略

## 四、产品二：情侣专属私密版

### 4.1 思考过程：为什么这样设计

**核心问题**：为什么情侣版不是"公开版的私密模式"，而需要独立设计？

**推理链条**：

1. **两个版本解决的核心问题完全不同**
   - 公开版解决："我知道该做但就是不做" → 动力问题
   - 情侣版解决："我们一起生活但总是忘/乱/推诿" → 协同问题
   - 前者靠"社会压力"，后者靠"情感连接"——底层驱动力不同

2. **情侣APP市场验证了高价值**
   - 125亿市场，26%增速，14.3%付费转化率（是效率工具的3倍+）
   - D30留存68.3%——用户一旦绑定就很难迁移
   - 但现有情侣APP（小恩爱、微光）都缺乏任务管理能力
   - **机会**：做"有任务管理能力的情侣工具"

3. **情侣场景的独特需求**
   - 不是"效率"，而是"仪式感"和"共同记忆"
   - 约会计划、纪念日、共同目标（存钱/健身/旅行）
   - 需要的是"一起完成"的感觉，而不是"谁做得更快"

4. **商业价值**
   - 情侣用户付费意愿极高（为仪式感买单）
   - 强绑定关系 = 极低流失率 = 高LTV
   - 可以作为独立产品或高级会员模块

### 4.2 产品定位

> **一句话定位**：情侣之间的"生活协同+情感陪伴"工具——一起规划、一起完成、一起记住。

### 4.3 功能设计

| 功能模块 | 具体功能 | 设计理由 |
|---------|---------|---------|
| **双人绑定** | 唯一情侣账号绑定、专属主页、亲密天数统计 | 仪式感的基础 |
| **共享清单** | 约会计划/旅行攻略/居家琐事/共同目标，双向可见，实时同步 | 核心协同功能 |
| **单人任务可选共享** | 个人任务可选择性对伴侣可见 | 不是全公开，保留个人空间 |
| **情侣专属任务类型** | 约会打卡、纪念日提醒、双向心愿清单、共同Flag | 区别于普通todolist的场景化设计 |
| **轻情绪互动** | 完成任务→对方收到暖心通知/鼓励特效/小表情 | 情感连接，不是效率驱动 |
| **双人AI复盘** | 每周/月生成"我们的时光报告"——共同完成了什么、下次想去哪 | 沉淀共同记忆 |
| **纪念日系统** | 自动记录关系里程碑、倒计时、专属提醒 | 情侣刚需 |
| **隐私隔离** | 情侣模块完全独立，零公开曝光，端到端加密 | 信任基础 |

**与公开版的核心差异**：

| 维度 | 公开版 | 情侣版 |
|------|--------|--------|
| 核心驱动 | 社会压力→完成任务 | 情感连接→共同生活 |
| 社交范围 | 公开/半公开 | 仅两人 |
| 交互调性 | 效率、竞争、成就感 | 温暖、陪伴、仪式感 |
| 数据流 | 可分享到社交平台 | 完全私密 |
| 变现逻辑 | 会员订阅 | 会员+仪式感付费（主题/特效） |

### 4.4 为什么不做"情侣+公开"的混合

- 情侣场景的核心是"私密安全感"，一旦有公开元素就会破坏信任
- 两个版本的用户心智完全不同，混合会导致定位模糊
- 技术上可以做"同一App内切换模式"，但建议先独立验证

## 五、盈利模型设计

### 5.1 思考过程：盈利点在哪

**推理链条**：

1. **效率工具付费渗透率低（4.3%），但AI功能能大幅提升付费率**
   - Todoist的AI功能用户升级率是基线5倍
   - 说明用户愿意为"AI帮我省时间"付费

2. **社交功能是最好的留存引擎，但不是直接的盈利点**
   - 社交提升粘性→粘性提升付费意愿→但社交本身不应收费
   - 收费会阻碍社交传播（"我想公开承诺但要付钱？算了"）

3. **情侣场景付费意愿极高（14.3%），且为"仪式感"愿意溢价**
   - 情侣用户不在乎"值不值"，在乎"有没有心意"
   - 主题/特效/纪念日定制是高毛利产品

4. **广告是最快变现但最容易伤害体验的方式**
   - 效率工具用户对广告极度敏感（"我花钱买效率，你还给我看广告？"）
   - 如果做广告，必须是"用户主动选择"模式（看广告换权益）

### 5.2 盈利方案设计

#### 公开版盈利模型

| 层级 | 定价 | 内容 | 目标 |
|------|------|------|------|
| **免费版** | ¥0 | 基础任务管理 + 每日3次AI + 3个公开承诺 + 基础社交 | 拉新+验证核心假设 |
| **Pro会员** | ¥68/年 | 去广告 + 无限AI + 高级复盘模板 + 专属主题 + 无限公开承诺 | 核心利润来源 |
| **广告收入** | -- | 看广告补打卡/解锁单次高级复盘/榜单展示位（用户主动选择） | 免费用户变现 |

**定价逻辑**：
- 不做学生单独定价——管理成本高，学生也会"假装不是学生"
- ¥68/年 = ¥5.7/月，低于一杯奶茶，决策成本极低
- 用"邀请3个好友注册送3个月Pro"替代学生价——社交裂变比低价更有价值

**收入预测模型**（保守估计）：
- 10万DAU → 5%付费率 → 5000付费用户 × ¥68 = ¥34万/年
- 广告收入：10万DAU × 10%看广告 × ¥0.5/次 × 3次/月 = ¥18万/年
- 第一年合计约¥50万（保守），如果社交裂变效果好可以更快

#### 情侣版盈利模型

| 层级 | 定价 | 内容 |
|------|------|------|
| **基础版** | ¥0 | 共享清单 + 纪念日 + 基础互动 |
| **情侣会员** | ¥128/年（双人） | 双人AI复盘 + 专属主题/特效 + 纪念日定制 + 时光报告 |
| **单品付费** | ¥6-30/件 | 限定主题包、纪念日特效、AI情书生成 |

**情侣版定价逻辑**：
- ¥128/年双人 = ¥64/人/年，比公开版Pro略低（因为功能更聚焦）
- 单品付费是情侣版的特色——为仪式感买单的意愿很强
- 节日营销（情人节/七夕/520）可以做限时主题，拉高收入峰值

### 5.3 盈利优先级

1. **先不考虑盈利**（MVP阶段）：专注验证核心假设和积累用户
2. **先上Pro会员**（验证后）：最简单、最不伤体验的变现方式
3. **再加广告**（有规模后）：用户主动选择型广告，不破坏核心体验
4. **最后做情侣版**（公开版成功后）：作为独立产品线或高级模块

## 九、风险与应对

| 风险 | 严重度 | 概率 | 应对策略 |
|------|--------|------|---------|
| 公开承诺在中国文化中接受度低 | 🔴 高 | 中 | MVP快速验证；设计"匿名公开"选项；从小圈子（好友可见）开始 |
| 社交冷启动无人互动 | 🔴 高 | 高 | 预置AI鼓励+官方运营种子社区；设计"无需好友也能玩"的赛道排行 |

| AI成本超预期 | 🟠 中 | 低 | 免费版限制AI次数；用国产模型降成本 |

## 十一、总结

### 可行性判断：条件性Go ✅

**成立条件**：
1. 公开承诺+轻社交在中国学生群体中被验证有效（MVP阶段验证）
2. 工具底座体验足够好（不输滴答清单核心功能）
3. AI能力成本可控（用国产模型）
4. 社交冷启动有可行方案

**最大风险**：
- 中国文化对"公开目标"的接受度是否足够（需要MVP验证）
- 社交冷启动的難度（需要运营投入）

**最大机会**：
- 没有任何头部产品同时做好"AI+社交问责"
- 行为经济学证据强烈支持"公开承诺提升完成率"
- 情侣版市场高增长+高付费意愿，且缺乏"任务管理+情感陪伴"的组合

**一句话结论**：
> 产品方向正确，市场空白明确，但需要极度聚焦MVP验证核心假设——"公开承诺+轻社交是否真的提升完成率"。先做好这一个点，再扩展其他功能。不要一上来就做全能产品。
